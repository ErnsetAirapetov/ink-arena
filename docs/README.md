# docs — документация InkArena

Здесь живёт всё, что не код: замысел игры, дизайн, планы. Код и баланс
документируют сами себя (`src/`, `src/config.ts`).

## Карта

```
docs/
├── README.md              ← ты здесь: карта и правила
│
├── design/                ← ФАЗА 0: проработка идеи (кухня, сырьё)
│   ├── questions.md       ← рамка: вопросы (волна 1 + волна 2 по ответам)
│   ├── answers.md         ← сырые ответы своими словами
│   ├── braindump.md       ← свободный поток идей вне рамки
│   ├── decisions.md       ← журнал закреплённых решений (Р1, Р2, ...)
│   ├── discussion.md      ← открытые темы на брейншторм (Т1, Т2, ...)
│   ├── proposals.md       ← черновые решения Клода по открытым вопросам
│   │                        (владелец помечает ✅/✏️, потом → decisions.md)
│   └── handoff.md         ← состояние работы: где остановились, что дальше
│
├── game/                  ← GDD: источник правды об игре (витрина)
│   │                        рождается из design/, файл за файлом
│   ├── vision.md          ← концепт на одну страницу: фантазия, игрок,
│   │                        уникальность, анти-скоуп
│   ├── spellcraft.md      ← ЯДРО: грамматика рисования — глифы-слова,
│   │                        правила композиции, стихии, чернила, осечки
│   ├── combat.md          ← дуэль: темп, чтение противника, победа,
│   │                        форматы матча
│   ├── modes.md           ← режимы вокруг дуэли: ранги, ПвЕ, турниры
│   ├── progression.md     ← мета между боями: прогрессия, экономика,
│   │                        честность соревнования
│   ├── ux.md              ← экраны, зона рисования, онбординг, реплеи
│   └── world.md           ← сеттинг, тон, визуальный стиль, референсы
│
├── tech/                  ← технический дизайн новой версии
│   │                        (появится после ядра game/; не раньше)
│   ├── architecture.md    ← симуляция/ввод/рендер, мультиплатформа
│   ├── recognition.md     ← распознавание и парсинг композиции
│   └── netcode.md         ← сеть, если/когда дойдём
│
└── prototype/             ← архив первого прототипа (код в src/ пока жив)
    ├── architecture.md
    └── spells.md
```

Файлы в `game/` и `tech/` НЕ создаются заранее — каждый рождается, когда
в `decisions.md` накопилось достаточно закреплённого для него. Первым появится
`game/vision.md`, следом `game/spellcraft.md` (по итогам темы Т1 — это самый
блокирующий документ: на грамматике стоит всё остальное).

## Конвейер: как сырьё становится продуктом

```
answers.md + braindump.md          (сырьё: ответы, идеи, поток)
        │
        ▼  разбор (Клод читает, находит решения и противоречия)
decisions.md + discussion.md       (закреплено / вынесено на обсуждение)
        │
        ▼  брейншторм-сессии по темам Т1, Т2, ...
decisions.md пополняется           (развилки закрыты)
        │
        ▼  кристаллизация
game/*.md                          (источник правды об игре)
        │
        ▼  только после ядра game/
tech/*.md → код                    (как это строим)
```

Роли файлов:
- **questions.md** — задаёт направление; пополняется волнами по мере ответов.
- **answers.md, braindump.md** — пишет автор, никто не правит задним числом.
- **decisions.md** — только решения с обоснованием; изменение решения — новая
  запись «пересмотрено», а не правка старой.
- **discussion.md** — живой список: тема закрыта → решение уезжает в
  decisions.md, тема вычёркивается.
- **game/** — единственное место, куда смотрим при разработке. Если game/
  противоречит design/ — прав game/ (design — черновики).

## Правила

- Один файл — одна тема. Разросся — делим.
- Сырое (answers, braindump) не чистим задним числом — это источник, а не витрина.
- Устаревшее не удаляем молча: либо в архив (как `prototype/`), либо пометка
  в шапке «устарело, см. X».
- Решение без «почему» — не решение. В decisions.md и game/ у каждого выбора
  есть обоснование.
