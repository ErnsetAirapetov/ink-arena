# docs — документация InkArena

Здесь живёт всё, что не код: замысел игры, дизайн, планы. Код и баланс
документируют сами себя (`src/`, `src/config.ts`).

## Структура

```
docs/
├── README.md            ← ты здесь: карта папки и правила работы с ней
├── design/              ← проработка идеи игры (фаза 0 — идёт сейчас)
│   ├── questions.md     ← рамка: вопросы, на которые должен ответить дизайн
│   ├── answers.md       ← ответы на вопросы — сырые, своими словами
│   └── braindump.md     ← свободный поток идей вне рамки вопросов
└── prototype/           ← архив первого прототипа (код в src/ пока жив)
    ├── architecture.md  ← как устроен код прототипа
    └── spells.md        ← шпаргалка по глифам и комбо прототипа
```

## Как это работает

**Сейчас — фаза 0: понять, что за игру мы делаем.**

1. `design/questions.md` — фиксированный список вопросов. Он и есть «рамка»:
   пока на ключевые вопросы нет ответов, к коду не возвращаемся.
2. `design/answers.md` — сюда пишутся ответы. Не надо красиво: пиши как думаешь,
   структурирование — отдельный шаг.
3. `design/braindump.md` — всё, что не ложится в вопросы: образы, механики,
   референсы, «а что если». Ни одна идея не должна потеряться потому,
   что для неё не нашлось графы.

**Дальше — из сырья в продукт.** Когда ответы и поток накопятся, из них
выкристаллизуются структурированные документы (они появятся в `design/` рядом):

- `vision.md` — концепт-документ: что за игра, для кого, почему в неё играют.
  Одна страница, источник правды для всех решений.
- `mechanics.md` — ядро геймплея: бой, заклинания, ресурсы, грамматика глифов.
- `progression.md` — удержание: прогрессия, мета, контент.
- `roadmap.md` — новая дорожная карта: от концепта к вертикальному срезу.

Не создаём эти файлы заранее — каждый рождается, когда для него есть содержание.

## Правила

- Один файл — одна тема. Разросся — делим.
- Сырое (answers, braindump) не чистим задним числом — это источник, а не витрина.
- Устаревшее не удаляем молча: либо в архивную папку (как `prototype/`),
  либо помечаем в шапке файла «устарело, см. X».
- Решение принято — оно попадает в структурированный документ со «почему»,
  а не остаётся жить только в потоке сознания.
