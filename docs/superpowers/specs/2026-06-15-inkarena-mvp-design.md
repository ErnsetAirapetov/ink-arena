# InkArena MVP — «Песочница глифов с комбо» (дизайн)

**Дата:** 2026-06-15
**Статус:** утверждён, готов к планированию реализации

## 1. Контекст и цель

InkArena — задумка про PvP-битвы заклинаниями, которые игроки **рисуют**.
Чем точнее нарисован глиф, тем сильнее заклинание; чем быстрее — тем быстрее
каст; разные глифы сочетаются в комбо. Вдохновение — аниме «академия колдовских
колпаков».

Это первый шаг (MVP), задача которого — **проверить, кайфово ли само ядро**:
свободно рисовать глифы, видеть точность/скорость, получать эффект и складывать
глифы в комбо. Без соперника, без онлайна.

**Критерий успеха MVP:** автор и несколько друзей открывают страницу по ссылке,
рисуют глифы, и это вызывает желание играть дальше («хочу драться этим»).

## 2. Платформа и стек

- **Платформа:** браузер (работает и мышью на ПК, и пальцем/стилусом на тач).
  Раздаётся по ссылке, ничего устанавливать не нужно.
- **Стек:** TypeScript + HTML5 Canvas + Vite (сборка/дев-сервер). Без игрового
  движка — для песочницы рисования нативный canvas даёт максимум прозрачности и
  скорости до результата.
- **Распознавание:** алгоритм **$P Point-Cloud Recognizer** (классический,
  бесплатный, ~сотня строк). Игрок рисует свободно, алгоритм находит ближайший
  эталон из внутреннего набора и возвращает оценку похожести `0..1`.
- **Python** в MVP не используется. Он запланирован как серверная часть
  (FastAPI: матчмейкинг, PvP-сессии, рейтинг) на более позднем этапе с онлайном.

## 3. Архитектура (файловая структура)

```
ink-arena/
├─ index.html              # холст + минимальный UI
├─ src/
│  ├─ main.ts              # точка входа, игровой цикл, связывает модули
│  ├─ drawing/             # ввод: палец/мышь → штрих (точки + время)
│  │  ├─ stroke.ts         # тип Stroke, запись точек down/move/up
│  │  └─ canvas-renderer.ts# отрисовка живого следа чернил
│  ├─ recognition/         # распознавание глифов
│  │  ├─ recognizer.ts     # алгоритм $P: match(точки) → {глиф, score}
│  │  └─ glyphs.ts         # эталоны: огонь, вода, щит, стрела
│  ├─ spells/              # глиф + точность + скорость → заклинание
│  │  ├─ spell-system.ts   # сборка Spell из результата распознавания
│  │  └─ combo.ts          # буфер последовательности → комбо
│  ├─ effects/             # простые частицы/вспышки на canvas
│  │  └─ effects.ts
│  ├─ ui/                  # элемент, % точности, скорость, состояние комбо
│  │  └─ hud.ts
│  └─ config.ts            # все «крутилки» баланса в одном месте
├─ tests/                  # юнит-тесты (Vitest)
├─ docs/                   # описание проекта, идея, дизайн
└─ CLAUDE.md               # инструкции по проекту для Claude Code
```

**Принцип:** каждый модуль делает одно понятное дело и общается через простой
интерфейс. Так проще читать человеку и надёжнее править ассистенту.

### Границы модулей (что делает / что отдаёт)

- **drawing** — превращает события указателя в `Stroke` (массив точек `{x, y, t}`).
  Не знает ничего про глифы. Отдаёт готовый штрих по завершении.
- **recognition** — чистая функция: `match(points) → { glyph, score }`. Без
  состояния, без DOM. Легко тестируется.
- **spells** — из `{ glyph, score }` и длительности штриха собирает `Spell`.
  `combo` хранит короткий буфер последних глифов и решает, получилось ли комбо.
- **effects** — рисует визуальный отклик на canvas. Не влияет на логику.
- **ui/hud** — показывает текстовую обратную связь (элемент, % точности,
  скорость, состояние комбо).
- **config** — числовые константы баланса (формулы силы/скорости, окно комбо,
  пороги). Меняешь баланс здесь, не трогая логику.

## 4. Поток данных

1. Палец/мышь: down → move… → up. `drawing` собирает `Stroke` (точки + время).
2. Конец штриха → `recognizer.match(stroke.points)` → `{ glyph, score 0..1 }`.
3. `длительность = points[last].t − points[0].t` → скорость каста.
4. `spell-system` собирает `Spell { element, power, castSpeed }`.
5. `combo`: если следующий глиф нарисован в окне ~2 сек → ищем пару в таблице
   комбо. Нашли → составное заклинание; нет → срабатывает одиночное.
6. `effects` + `hud` отображают результат.

## 5. Баланс и контент (всё в `config.ts`)

- **Сила:** `power = round(score * 100)` → диапазон 0–100.
- **Скорость:** пороги длительности штриха → `fast` / `normal` / `slow`.
- **Стартовый набор глифов (намеренно крошечный):**
  - `△` огонь
  - `~` вода
  - `○` щит
  - `/` стрела
- **Стартовые комбо (последовательность из 2 глифов):**
  - огонь + стрела → огнешар
  - вода + щит → лечащий барьер

Набор маленький специально: хватает почувствовать систему комбо, и легко
добавлять новые глифы/комбо позже.

## 6. Комбо-модель

**Последовательная, в окне времени.** Игрок рисует глиф A, затем глиф B в
течение окна (~2 сек, константа в `config.ts`). Пара ищется в таблице комбо:
найдена → составное заклинание; не найдена или окно истекло → срабатывает
одиночное заклинание последнего глифа. Одновременного наложения глифов в MVP
нет — это возможное расширение позже.

## 7. Тестирование

- **Vitest** юнит-тестами покрываем логику без UI:
  - `recognition`: известные наборы точек → ожидаемый глиф и разумный score.
  - `combo`: последовательности глифов → ожидаемое составное заклинание (и
    корректное поведение при истёкшем окне).
  - `spell-system`: проверка формул силы/скорости на граничных значениях.
- **Рисование, эффекты, ощущение** проверяются вручную в браузере (это и есть
  главная цель MVP — субъективный «кайф»).

## 8. Вне области MVP (YAGNI)

Сознательно НЕ делаем сейчас:

- соперник, HP, условие победы;
- сеть, онлайн-PvP, матчмейкинг;
- аккаунты, рейтинг, сохранения;
- звук;
- Python-сервер;
- одновременное наложение глифов;
- мобильная упаковка/нативные сборки (работает как веб-страница в браузере).

Всё перечисленное — кандидаты на следующие шаги после проверки идеи.

## 9. Дополнительные артефакты проекта

В рамках MVP также создаём:

- `README.md` — описание проекта (что это, как запустить, как тестировать,

  ссылкой. Конкретную площадку выбираем на этапе плана.

документация по разработке, идеи фич, планы и т.д. Минимальная структура:

```
docs/
├─ idea.md            # описание идеи игры (видение, механики, вдохновение)

```
