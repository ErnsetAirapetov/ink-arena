# tech/architecture — новый движок InkArena

Детальный тех-дизайн кодовой базы вехи 1 (MVP). Принципы — в корневом
`../architecture.md` (Р38); здесь — как они превращаются в пакеты, модули и
правила. Прототип в `src/` в новую базу не входит: сносится начисто при
закладке monorepo (Р43), код пишется с нуля по спекам `../game/`.

---

## Monorepo: npm workspaces

Инструмент — **npm workspaces** (Р44): при двух-трёх пакетах и одном
разработчике turborepo/nx — лишний слой. Пересмотр возможен, когда появится
сервер и станет тесно.

```
package.json            ← workspaces: ["packages/*"]
packages/
├── core/               ← детерминированное ядро (без DOM и Canvas)
│   └── src/
│       ├── config.ts   ← ВСЕ числа баланса (единственное место)
│       ├── sim/        ← симуляция боя
│       ├── spellcraft/ ← словарь глифов, парсер композиции, cost-модель
│       ├── recognition/← $P point-cloud, нормализация штрихов
│       └── geometry/   ← примитивы: точки, штрихи, bounding box, длины
└── client/             ← веб-клиент (Vite + Canvas)
    └── src/
        ├── render/     ← арена, снаряды, эффекты
        ├── input/      ← pointer events → штрихи (палец прежде мыши, Р13)
        ├── ui/         ← блокнот мага, HUD, экраны (game/ux.md)
        └── screens/    ← кампания, бой, результаты
```

`server/` появится к альфе (веха 2) поверх того же `core` — сейчас пакет не
создаётся (файлы не создаются заранее — правило docs/README).

## core: детерминированная симуляция

Ядро — чистая функция `step(state, inputs, dt) → state` с фиксированным
тиком. Из этого следуют жёсткие правила (нарушение — блокер на ревью):

1. **Запрещены в core:** `Date.now()`, `performance.now()`, `Math.random()`,
   DOM/Canvas/window, любые side effects и таймеры. Время существует только
   как номер тика; случайность — только seeded RNG из состояния.
2. **Состояние сериализуемо.** Всё состояние боя — plain-объекты; снапшот
   в JSON в любой момент. Это фундамент реплеев, тестов и будущего сервера.
3. **Реплей = сид + журнал входных событий.** Вход: штрихи игрока с
   таймстампами-тиками, касты, системные события раунда. Воспроизведение —
   прогон того же журнала через `step` (Р12, Р38).
4. **События наружу.** core сообщает клиенту о происходящем списком событий
   тика (каст, попадание, осечка, конец раунда) — рендер подписывается,
   ядро о нём не знает.

Модули core и их источники правды:

| Модуль | Что делает | Спека |
|---|---|---|
| `geometry` | штрихи, ресемплинг, длины, bbox, кластеризация | — (математика) |
| `recognition` | $P: эталоны глифов → распознавание с точностью | game/spellcraft.md |
| `spellcraft` | парсер «ядро и орбиты» (Р26), валидность/осечка, cost-модель чернил, формула силы | game/spellcraft.md |
| `sim` | HP, чернила/реген, снаряды и полёт (f(размер)), пентаграмма, статусы, раунды и матч (Р22, Р28, Р41), бот-манекен ПвЕ | game/combat.md |

Порядок каста внутри тика: штрихи → `recognition` (глифы + точность) →
`spellcraft` (композиция → валидное заклинание или осечка) → `sim`
(событие каста → снаряд/эффект). Клиент в этой цепочке не участвует —
он только собирает штрихи и рисует события.

## client: тонкий слой

Клиент не содержит игровых правил. Его обязанности: собрать штрихи
(pointer events, одинаково для пальца/мыши/стилуса), показать состояние
(Canvas-рендер по событиям core), экраны и блокнот мага по `game/ux.md`.
Разрешение и зоны — от макета Р10/Р35 (арена ~70%, блокнот — нижняя треть).
Клиент можно ломать и переделывать, не трогая core, — и наоборот.

## Тесты

- **core — только TDD** (Vitest): каждый модуль рождается тестом.
- **Golden-тесты распознавания:** корпус эталонов Р32 (пять заклинаний +
  их искажённые варианты) — постоянный регрессионный набор: «Искра
  распознаётся, две стихии — осечка, глиф на границе — осечка».
- **Симуляция:** сценарные тесты «журнал входа → ожидаемое состояние»
  (детерминизм делает их тривиальными) + инвариант-тесты (HP не растёт от
  урона, чернила в [0, max], раунд всегда разрешается по Р41).
- **client** — руками в браузере, как и раньше (соглашение CLAUDE.md).

## Сборка и инструменты

TypeScript строгий (`strict: true`) везде. `core` собирается tsc,
`client` — Vite (dev-сервер и бандл), core подключается как workspace-пакет.
Vitest на весь repo одним конфигом. PWA/Capacitor — упаковка клиента, к
бете (Р38), сейчас не настраивается.

## Миграция: снос прототипа (Р43)

В PR закладки monorepo, атомарно:
1. тег `prototype-archive` на последний коммит с `src/` (история остаётся);
2. `src/`, `tests/`, старые vite/tsconfig корня — удаляются;
3. корневые скрипты `npm run dev/test/build` переключаются на workspaces
   (поведение команд для агентов не меняется);
4. CLAUDE.md/AGENTS.md: путь баланса меняется на
   `packages/core/src/config.ts`, упоминание «прототип будет переписан»
   убирается.
Никакого переноса кода из прототипа — включая $P: реализуется заново по
TDD против golden-корпуса (Р43). `docs/prototype/` остаётся архивной докой.

## Порядок первых работ (эскиз для груминга)

1. Закладка monorepo + снос прототипа (пустые пакеты, тесты и dev-сервер
   зелёные) — одна задача, один PR.
2. `geometry` + `recognition`: $P и нормализация против golden-корпуса.
3. `spellcraft`: парсер композиции и осечки (правила Р26), cost-модель.
4. `sim`: раунд с ботом-манекеном — чернила, снаряды, пентаграмма, Р41.
5. `client`: ввод + блокнот + рендер арены — первый играбельный билд.
   ← здесь проверяется риск Р22 (рисование под давлением), см. Р42.
6. Кампания-головоломки (Р33) поверх работающего боя.

## Открытые вопросы

- Формат эталонов глифов (точки в коде / JSON-ассеты) — решится в задаче 2.

  баланса (Р30, config), архитектуре не мешает.
- Хранение прогресса кампании (localStorage достаточно для MVP) — решить
