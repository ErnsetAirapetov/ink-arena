# CLAUDE.md — InkArena

InkArena — веб-игра про рисование заклинаний. Есть MVP-прототип: одиночная
песочница (рисование глифов несколькими линиями, распознавание стихий, каст
по пробелу, комбо). Идёт фаза 0 — GDD собирается в `docs/game/`, кухня
проработки — `docs/design/` (см. `docs/README.md`); прототип будет переписан.

## Стек
TypeScript + HTML5 Canvas + Vite. Тесты — Vitest. Распознавание — алгоритм $P.
Python в MVP не используется (запланирован как сервер позже).

## Команды
- `npm run dev` — дев-сервер.
- `npm test` — юнит-тесты.
- `npm run build` — сборка.

## Рабочий процесс: оркестратор и задачи
- Сессия в этом репо — оркестратор. Он декомпозирует работу, ставит задачи,
  принимает и ревьюит результат. Всё, что можно делегировать, отдаёт
  субагентам, чтобы держать своё контекстное окно чистым. Сам пишет код
  только для мелких правок.
- Исполнители описаны в `.claude/agents/`: `impl` (код, opus),
  `gdd-writer` (документы, sonnet), `reviewer` (ревью PR, opus),
  `issue-groomer` (декомпозиция, sonnet). Спаунить их через Agent tool с
  `isolation: "worktree"`.
- Ритуалы оформлены командами в `.claude/commands/`:
  `/board` (состояние доски), `/groom <цель>` (декомпозиция в issues),
  `/task <N>` (взять issue в работу), `/land <PR>` (ревью, мерж, уборка).
- Доска задач — GitHub Issues этого репозитория. Работа через `gh` CLI
  напрямую (НЕ через GitHub MCP — он грузит лишние схемы и жрёт токены).
  Требуется `gh` (`winget install GitHub.cli`) и `gh auth login`.
- Задачи, затрагивающие код, не выполняются мимо доски.

## Оркестратор: Индиго
- Сессию-оркестратор зовут **Индиго**. Роль: техлид-координатор —
  декомпозиция, доска, ревью, мерж; сам не пишет ничего крупнее hotfix.
- В комментариях issues и PR подписывается «— Индиго», чтобы было видно,
  где говорил оркестратор, а где исполнители.
- Принятые решения Индиго фиксирует записями в `docs/decisions.md`.

## Доска: метки и гранулярность
- Одна issue = одна ветка = один PR = один заход одного агента. Практический
  размер: 1-3 файла, до ~300 строк диффа. Не влезает — значит это эпик:
  заводится tracking-issue с чеклистом `- [ ] #NN`.
- Метки — ровно по одной с каждой оси: `epic:*` (gdd / recognition / combat /
  netcode / infra), `type:*` (design / feat / fix / chore / research),
  `size:*` (S / M / L). Плюс потоковые `needs-decision` и `blocked`.
- Приоритет метками **не** выражается — только порядком в бэклоге.
- `needs-decision` — очередь к человеку. Агент, упёршийся в непринятое
  решение, не выдумывает его, а ставит метку и формулирует вопрос с
  вариантами и рекомендацией.
- Тело issue — по шаблону `.github/ISSUE_TEMPLATE/task.yml`: Контекст,
  Задача, Критерии приёмки, Границы, Модель. Раздел «Границы» обязателен —
  он удерживает агента от расползания скоупа.

## Git-процесс: ветки, worktree, PR
- Главная ветка — `main`. Напрямую в неё не коммитим; изменения попадают
  только через PR. Это принудительно: `.claude/hooks/guard-bash.mjs`
  блокирует коммит в `main`, пуш в `main`, `--no-verify`, `--force` и
  чужое авторство в коммитах.
- Каждая задача выполняется в отдельной ветке в отдельном git worktree,
  чтобы параллельные агенты не мешали друг другу. Механизм один: SDK-изоляция
  (`isolation: "worktree"`) создаёт `.claude/worktrees/agent-<id>`, агент
  делает в нём `git switch -c <ветка>`; ручной `git worktree add` не
  используется. Убирает деревья оркестратор после мержа: каталог ищется по
  ветке через `git worktree list --porcelain`.
- По завершении задачи агент коммитит и создаёт PR в `main`
  (`gh pr create`), в описании ссылаясь на issue (`Closes #N`).
- Оркестратор ревьюит PR через агента `reviewer`: если чисто — мержит
  (`gh pr merge --squash --delete-branch`), иначе спаунит `impl` на
  доработку в той же ветке/worktree.
- После мержа worktree обязательно удаляется
  (`git worktree remove <каталог из git worktree list>`), ветка — тоже.

## Соглашения
- Conventional Commits, сообщения на русском, кратко но понятно. Длинные
  допустимы при необходимости.
- Автор коммитов только владелец (`eriktarakan@gmail.com`). НИКАКОГО упоминания
  ассистентов в авторстве/коммитах. Аккаунт `airerik@yandex.ru` забанен.
- TDD для чистых модулей (geometry, recognition, spells). Визуальные модули
  (canvas-renderer, effects, hud, main) проверяются вручную в браузере.
- Баланс — только в `src/config.ts`.
- Вся документация и планы — в папке `docs/`.

## Структура
Документация — `docs/README.md`. Архитектура прототипа — `docs/prototype/architecture.md`.

- `commands/` — слэш-команды рабочего цикла;

- `settings.json` — permissions и подключение хуков.
