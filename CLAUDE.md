# CLAUDE.md — InkArena

InkArena — веб-игра про рисование заклинаний. Есть MVP-прототип: одиночная
песочница (рисование глифов несколькими линиями, распознавание 6 стихий, каст
по пробелу, комбо). Сейчас идёт фаза 0 — проработка идеи и полного описания
игры в `docs/design/` (см. `docs/README.md`); прототип будет переписан.

## Стек
TypeScript + HTML5 Canvas + Vite. Тесты — Vitest. Распознавание — алгоритм $P.
Python в MVP не используется (запланирован как сервер позже).

## Команды
- `npm run dev` — дев-сервер.
- `npm test` — юнит-тесты.
- `npm run build` — сборка.

## Рабочий процесс: оркестратор и задачи
- Сессия в этом репо — оркестратор (модель Fable). Он декомпозирует работу,
  ставит задачи, принимает и ревьюит результат. Всё, что можно делегировать,
  он отдаёт субагентам (сам их создаёт и дирижирует), чтобы держать
  контекстное окно основной сессии чистым. Сам пишет код только для мелких
  правок.
- Исполнение — субагенты через Agent tool с указанием модели:
  - `model: "opus"` — сложные задачи (архитектура, нетривиальный код, ресёрч);
  - `model: "sonnet"` — рутинные (механические правки, тесты по образцу, доки).
- Доска задач — GitHub Issues этого репозитория. Работа с ней через `gh` CLI
  напрямую (НЕ через GitHub MCP — он грузит в контекст лишние схемы
  инструментов и жрёт токены). Требуется установленный `gh` (`winget install
  GitHub.cli`) и `gh auth login`.
- Цикл задачи: оркестратор создаёт issue (заголовок — суть, тело — контекст и
  критерии приёмки) → спаунит субагента с номером issue в промпте → принимает
  результат → закрывает issue комментарием-итогом (`gh issue close N -c "..."`).
- Issues нужны для наблюдаемости: по ним виден порядок, статус и история
  работ. Задачи, затрагивающие код, не выполняются мимо доски.

## Git-процесс: ветки, worktree, PR
- Главная ветка — `main`. Напрямую в неё не коммитим; изменения попадают
  только через PR.
- Каждая задача выполняется в отдельной ветке в отдельном git worktree,
  чтобы параллельно запущенные агенты не мешали друг другу. Worktree живут
  в `.claude/worktrees/` (папка в `.gitignore`):
  `git worktree add .claude/worktrees/<ветка> -b <ветка>`.
- По завершении задачи агент коммитит и создаёт PR в `main`
  (`gh pr create`), в описании ссылаясь на issue (`Closes #N`).
- Оркестратор ревьюит PR: если всё ок — мержит (`gh pr merge --squash`),
  иначе спаунит агента на доработку в той же ветке/worktree.
- После мержа worktree обязательно удаляется
  (`git worktree remove .claude/worktrees/<ветка>`), ветка — тоже.
